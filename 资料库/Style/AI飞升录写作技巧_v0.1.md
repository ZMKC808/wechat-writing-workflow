# AI飞升录写作技巧 v0.1

来源样本：`/Users/zmkc808/Downloads/文章：AI飞升录`，共 93 篇公众号 Markdown。

用途：后续写作时，优先按本文判断选题、搭骨架、控制语言和结构。它不是最终 skill，等实战跑一篇后再固化。

---

## 1. 账号定位

AI飞升录不是泛 AI 新闻号，也不是纯教程号。

它更像一个“AI 工具与红利雷达”：帮读者更早发现 AI 新模型、新工具、新玩法、新福利、新风险，并尽量给到可执行路径。

读者画像：

- 已经在用 AI，但信息来源杂乱；
- 对 Claude Code、Codex、OpenClaw、Agent、模型 API、提示词、自动化工作流敏感；
- 喜欢“能马上试”的内容，不只想看行业宏大叙事；
- 能接受一点技术门槛，但不喜欢学院腔；
- 对“白嫖、限免、内测、低价、全网首发、实测、保姆级教程”反应强。

账号的核心承诺：

> 我替你盯 AI 前沿，把真正有用、有红利、有坑的东西拆出来，能上手的直接给路径，不能上手的给判断。

## 3. 选题判断标准

一个选题值得写，至少命中下面 3 项：

| 维度 | 通过标准 |
|---|---|
| 新 | 过去 24-72 小时内出现明显新信号，或中文圈还没充分写透 |
| 能试 | 读者看完能注册、部署、对比、领取、复现、避坑 |
| 有差异 | 不只是复述官方公告，有实测、反常识、成本对比、路径拆解 |
| 有红利 | 免费、限免、低价、内测、早期机会、信息差 |
| 有风险 | 封号、失效、政策、合规、隐私、成本、技术坑 |
| 有故事 | 人物、公司、封号、暴涨、翻车、反转、争议 |
| 有账号相关性 | 和 Claude Code、Codex、Agent、AI 工具、自动化、开发者红利有关 |

不建议写：

- 只有大厂通稿，没有实测入口；
- 只有情绪，没有可验证事实；
- 中文圈已经刷屏，但没有新角度；
- 需要太多背景知识，读者看完无法行动；
- 纯宏观 AI 口水战，无法落到工具、案例或判断。

选题评分：

```text
总分 10 分

时效性：0-2
可操作：0-2
信息差：0-2
账号相关：0-2
风险/争议/故事性：0-2

8 分以上：当天优先写
6-7 分：补资料后写
5 分以下：除非用户指定，否则不主动写
```

## 4. 标题技巧

标题不追求端庄，追求“信号强 + 点击理由明确”。

常见标题公式：

1. 时间/突发 + 主体 + 结果
   - 刚刚，Claude Code源码泄露了
   - OpenAI 深夜发布 GPT-5.4 mini 和 nano

2. 全网首发/实测 + 新东西
   - 全网首发，Anthropic 发布官方「龙虾」
   - GLM 5.2实测：把Claude、GPT、Kimi全拉进来比了一遍

3. 福利/紧迫感 + 具体收益
   - 抓紧薅！免费用1个月Claude4.7
   - 官方最低！法国区ChatGPT Team 59元上车

4. 反常识/冲突 + 问题
   - Claude Design杀死了Figma，前端的出路在哪？
   - 豆包的定价还能再高些

5. 结果夸张 + 解释型
   - 一年暴赚45倍，“白毛女”Serenity凭什么？
   - 3天54k Star！Karpathy让AI自己做科研去了

标题注意：

- 可以有强情绪，但不要滥用“炸了、炸裂、震惊”；
- 数字越具体越好：`30ms`、`54k Star`、`59元`、`1M token`；
- “刚刚、全网首发、实测、抓紧薅、保姆级”是高频有效词，但不能每篇都用；
- 技术教程标题要明确收益，不要只写工具名；
- 非技术热点标题要有冲突或人物，不要写成资讯摘要。

## 5. 开头结构

AI飞升录的开头很少铺背景，常见是直接给新信号。

有效开头有 5 种：

### 5.1 直接爆点

适合新模型、新工具、新福利。

```text
讯飞星辰现在有个活动，注册就能用 Qwen3.6 和 Qwen3.5，Token 不设上限。
6 月 30 日截止，现在进来还来得及。
```

写法：第一句给发生了什么，第二句给为什么现在要看。

### 5.2 个人遭遇

适合工具教程和方法论。

```text
上周我用 Claude Code 写了个数据处理脚本，两个小时交付。
打开一看：函数套函数，变量名 temp1、data2，同一段逻辑复制粘贴了三次，测试零个。
```

写法：先给一个具体场景，再转到问题。

### 5.3 先承认疑惑

适合模型实测和热点判断。

```text
我还纳闷，界面显示是 2.5 Pro，怎么可能变强。
后来我又去 X 上和小红书搜，越来越多人在晒对比图。
```

写法：不要一上来装全知，先写自己怎么发现问题。

### 5.4 事实 + 反差

适合人物和行业故事。

```text
她不是追热点出名的。恰恰相反，她专门挖那些被市场忽视的、藏在 AI 供应链最上游的隐形瓶颈。
```

写法：给读者一个反差，让他知道为什么这个人/事值得看。

### 5.5 结论先行

适合短福利、短通知。

```text
官方永久降价，比之前便宜 1/4。
这个性能这个价格完全可以。
```

写法：不讲故事，直接告诉读者收益。

## 6. 正文结构

不要默认用传统 SEO 的 H1/H2/H3。AI飞升录更像公众号阅读流，结构靠短小标题、图片、案例和步骤推进。

通用骨架：

```markdown
开头：新信号 / 个人遭遇 / 结论

小标题一：具体怎么回事
解释核心事实，放第一张关键截图。

小标题二：我试了下
写实测、对比、数据、失败点。

小标题三：问题在哪
拆底层逻辑，不只复述公告。

小标题四：怎么上手
给路径、步骤、链接、注意事项。

小标题五：我的判断
给建议，不做空泛总结。
```

小标题规则：

- 最多 5-6 个；
- 优先 4-6 个字；
- 不写长问题式标题；
- 非技术文章不放 FAQ；
- 技术教程可以带 FAQ，也可以不带。只有当读者真的会遇到订阅、安全、费用、配置、失败排查时才加。

## 7. 技术文章模板

技术文章的核心不是“讲清楚概念”，而是“让读者少踩坑”。

结构建议：

```markdown
开头：一句话说明结果和收益

适合谁
说明前提，不符合条件的人别折腾。

准备什么
账号、系统、订阅、服务器、域名、费用。

开始操作
按真实路径写，不省略关键界面。

容易卡住
写权限、网络、端口、订阅、模型、区域、风控。

怎么验证
告诉读者成功标志。

怎么选择
轻量方案 vs 完整方案。

常见问题（可选）
只有技术/费用/安全类问题多时才加。
```

技术文章必须有：

- 具体数字：价格、时长、配置、端口、延迟、Token、版本；
- 成功标志：看到什么页面、返回什么结果、命令跑完是什么样；
- 失败路径：哪一步最容易卡，怎么判断是不是这个问题；
- 截图或命令：能截图就截图，不能截图就给命令。

技术文章可以保留 FAQ：

- 订阅要求；
- 安全性；
- 费用；
- 手机/Windows/Mac 是否支持；
- 失败排查；
- 国内网络可用性。

---

## 8. 非技术文章模板

非技术文章不要 FAQ，不要像百科词条。

结构建议：

```markdown
开头：人物/事件/反差

故事起点
什么时候发生，谁做了什么。

关键转折
为什么这件事不普通。

底层逻辑
拆到物理瓶颈、商业模式、政策约束、成本结构。

几个案例
用 2-3 个案例证明判断。

我的判断
一句话收，不展望未来。
```

非技术文章的重点：

- 用人物或事件带入；
- 用具体数字建立可信度；
- 用“往下拆”的方式形成深度；
- 不强行升华到时代、命运、文明；
- 不结尾 FAQ。

## 9. 语言风格

核心风格：短句、直接、口语、有判断、带一点个人反应。

高频有效表达：

- 我一开始没打算写这篇。
- 我点进去看了一下。
- 我试了一遍。
- 我看到第三条就坐不住了。
- 这个体验挺有代表性的。
- 问题出在这里。
- 好处是，坏处是。
- 怎么选。
- 够了。

句子规则：

- 一段通常 1-3 句；
- 长段只用于解释技术链路或案例；
- 每段尽量只有一个意思；
- 关键数字加粗；
- 少用“此外、值得注意的是、总而言之”；
- 少用“不是 A，而是 B”排比；
- 不使用“不是...而是...”结构；
- 不使用 `——` 符号；
- 正文不使用分割线；
- 可以写“我”，但必须绑定动作：我试了、我翻了、我看到、我卡住。

## 10. 深度来源

AI飞升录的深度不是靠堆概念，而是靠继续往下拆。

常见拆法：

1. 从模型到成本
   - 模型升级不只看跑分，要看 API 成本、上下文、工具调用、延迟、可用入口。

2. 从工具到工作流
   - 新工具不只看功能，要看它改变了哪一步流程。

3. 从热点到物理/商业约束
   - 例如 GPU 继续往下拆到光模块、衬底、材料、能源。

4. 从官方公告到真实使用
   - 公告说强，自己拿任务测；公告说免费，自己查限制。

5. 从教程到避坑
   - 真正有用的不是步骤，而是哪些步骤会失败。

写作时每篇至少要有一个“往下拆”的段落，否则容易变成资讯搬运。

## 11. 图片规则

当前样本平均每篇约 6.3 张图，中位数 4 张图。

不同类型图片密度：

- 短福利：2-4 张图；
- 模型实测：4-8 张图；
- 技术教程：8-30 张图；
- 深度精译：4-15 张图；
- 品牌叙事：1-7 张图。

图片作用：

- 第一张图证明“这事是真的”；
- 中间图展示效果、截图、对比、配置步骤；
- 技术教程图要跟步骤强绑定；
- 非技术文章图不要堆，保留关键证据即可。

后续从外部文章取图时，必须做图片清单：

| 图片 | 来源 | 用途 | 风险 | 处理 |
|---|---|---|---|---|
| 原文截图 | 公众号/X/官网 | 证据 | 中 | 少量引用或重绘 |
| 产品界面 | 官网/自测 | 说明 | 低-中 | 优先自测截图 |
| 人物/媒体图 | 媒体 | 氛围 | 高 | 尽量不用或 GPT 重绘 |
| 流程图 | 自制 | 解释 | 低 | 推荐 |

## 12. 发布前检查

写完后按这个顺序检查：

1. 这篇主要靠什么成立：新、福利、教程、实测、观点、故事？
2. 开头前 3 句有没有直接给信号？
3. 是否至少有 1 个“我试了/我查了/我对比了”的证据？
4. 是否有具体数字、具体限制、具体路径？
5. 小标题是否超过 6 个？
6. 小标题是否太长？
7. 非技术文章是否误加了 FAQ？
8. 技术文章如果有 FAQ，问题是否真有必要？
9. 是否有“首先、其次、总而言之、值得注意的是、说实话、翻译成人话、不吹不黑”？
10. 是否出现“不是...而是...”结构？
11. 是否出现 `——` 符号？
12. 是否出现分割线？
13. 结尾是否在空泛升华？

## 13. 后续写作流程

用户给一个公众号/X/媒体链接后，按这个流程走：

1. 读取原文
   - 抓标题、作者、时间、正文、图片、链接、核心观点。

2. 同题扩展
   - 找公众号、X、GitHub、官网、媒体、Reddit/社区讨论。

3. 拆素材
   - 分成事实、数据、观点、案例、争议、教程步骤、图片证据。

4. 判断角度
   - 不能和原文同结构；
   - 找一个新的切入点：实测、风险、成本、工作流、人物、底层约束。

5. 先出骨架
   - 最多 5-6 个小标题；
   - 小标题 4-6 个字；
   - 标注每节用哪些素材。

6. 写正文
   - 按 AI飞升录语气；
   - 少总结，多事实；
   - 技术文可选 FAQ，非技术文不加 FAQ。

7. 去 AI 味
   - 删高频 AI 词；
   - 打散规整排比；
   - 加入真实测试/具体限制；
   - 结尾一句话收。

8. 排版与发布
   - FlowCast 排版；
   - 图片清单；
   - GPT 生封面；
   - 上传公众号草稿；
   - 用户确认后发布。
